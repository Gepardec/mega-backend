## Context

The `com.gepardec.mega.hexagon` package is a greenfield hexagonal backend being built alongside the legacy `com.gepardec.mega` layered architecture. The first capability to build is the User concept, which is assembled from two upstream systems: ZEP (timesheet, primary) and Personio (HR, secondary). The legacy code already has working REST clients for both systems — this design wraps them rather than replacing them.

## Goals / Non-Goals

**Goals:**
- Define the `User` aggregate root with its own stable domain UUID
- Separate ZEP-sourced and Personio-sourced data into distinct value objects (`ZepProfile`, `PersonioProfile`)
- Implement the 30-minute sync use case using clean Ports & Adapters
- Keep the domain and application layers free of Quarkus and persistence annotations
- Wrap (not re-implement) the existing legacy ZEP and Personio REST clients in the outbound adapters

**Non-Goals:**
- Replacing or modifying any code in `com.gepardec.mega` (legacy package)
- Implementing `PROJECT_LEAD` role derivation (requires ZEP project fetch — deferred)
- Exposing any REST API for User in this change
- User management (create/update via API) — sync is the only write path

## Decisions

### D1: User gets its own domain UUID

**Decision**: `UserId` is a domain value object wrapping a `UUID` generated by our system on first sync, not derived from ZEP's username.

**Rationale**: ZEP usernames can change. Coupling our primary key to an external system's identifier creates instability. The domain UUID is stable forever; ZEP username is stored in `ZepProfile` as an external reference.

**Alternative considered**: Using ZEP username as the domain identifier — rejected because it creates a hard dependency on ZEP's identity scheme.

### D2: ZepProfile and PersonioProfile as nested value objects

**Decision**: The `User` aggregate contains `zepProfile: ZepProfile` (non-null after first sync) and `personioProfile: PersonioProfile` (nullable).

**Rationale**: Separating by provenance makes the data's origin explicit and prevents mixing fields from different systems. It also makes it easy to update one profile independently during sync.

**Alternative considered**: Flat field structure on `User` — rejected because it obscures which system owns which field.

### D3: Aggregate exposes sync methods

**Decision**: `User` exposes `syncFromZep(ZepProfile)` and `syncFromPersonio(PersonioProfile)` as domain methods. The use case calls these; it does not mutate aggregate fields directly.

**Rationale**: Keeps mutation logic inside the aggregate boundary. The aggregate can enforce invariants (e.g., not clearing an existing PersonioProfile if the new one is null).

### D4: UserSyncConfig as an application-layer record

**Decision**: Office management usernames are passed to `SyncUsersService` via a `UserSyncConfig` record, populated by the Quarkus adapter layer from `@ConfigProperty`.

**Rationale**: The use case must not import Quarkus annotations. Injecting config as a plain record keeps the application layer portable and testable without a CDI container.

### D5: Outbound adapters wrap legacy REST clients

**Decision**: `ZepEmployeeAdapter` and `PersonioEmployeeAdapter` delegate to the existing legacy `ZepEmployeeRestClient` and `PersonioEmployeesClient` respectively.

**Rationale**: The legacy REST clients are stable, tested, and handle auth/retry. Re-implementing them would be duplication. The adapters' sole job is translation: legacy DTO → hexagonal value object.

### D6: Sync match key is ZEP username

**Decision**: On each sync, existing `User` records are located by `ZepProfile.username` via `UserRepository.findByZepUsername(String)`.

**Rationale**: ZEP username is the only stable external identifier available before a User is created. Email could change; Personio ID is not available until after Personio lookup.

### D7: Personio enrichment is best-effort

**Decision**: If `PersonioEmployeePort.findByEmail()` returns empty (Personio unavailable or no match), the existing `personioProfile` on the User is preserved. No error is thrown; the sync continues.

**Rationale**: Personio data is supplementary. A Personio outage should not block the sync of ZEP employee data.

## Sync Flow

```
UserSyncScheduler (@Scheduled PT30M)
    │
    ▼
SyncUsersUseCase.sync()
    │
    ├─1─▶ ZepEmployeePort.fetchAll()
    │       → List<ZepProfile>
    │
    ├─2─▶ UserRepository.findAll()
    │       → Map<zepUsername, User>
    │
    ├─3─▶ for each ZepProfile:
    │       if user exists → user.syncFromZep(zepProfile)
    │       if not         → User.create(newUUID, zepProfile, EMPLOYEE role)
    │       if username in UserSyncConfig → add OFFICE_MANAGEMENT role
    │
    ├─4─▶ for each User:
    │       PersonioEmployeePort.findByEmail(user.email)
    │         if present  → user.syncFromPersonio(personioProfile)
    │         if absent   → preserve existing personioProfile (best-effort)
    │
    ├─5─▶ deactivate Users whose ZEP username is no longer in ZEP response
    │
    └─6─▶ UserRepository.saveAll(users)
```

## Risks / Trade-offs

- **PersonioProfile nullability** → Some features may need null-checks on `personioProfile`. Mitigated by making callers explicitly handle the Optional/null case.
- **Wrapping legacy clients** → The hexagonal adapters inherit any bugs or limitations of the legacy REST clients. If a legacy client changes its interface, the adapter must update. Mitigated by keeping adapter translation logic thin.
- **No PROJECT_LEAD role** → Users who are project leads in ZEP will not have that role until a future change adds project sync. Accepted as a known gap.
- **Sync concurrency** → If two scheduler instances run (e.g., dev + test), double-sync could occur. Mitigated by Quarkus scheduler's single-node guarantee in dev mode; production uses a single instance.
